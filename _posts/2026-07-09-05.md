### Entity とは

`@Entity` はDBのテーブルと1対1で対応するオブジェクトです。JPA（Hibernate）の永続化コンテキストによって状態が管理され、フィールドの変更がそのままUPDATEクエリに反映されることもあります（ダーティチェッキング）。

```java
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String email;
    private String password; // DBにしか存在しない値

    @OneToMany(mappedBy = "user")
    private List<Order> orders;
}
```

### DTO とは

DTO（Data Transfer Object）は、層と層の間でデータを運ぶためだけのシンプルな入れ物です。DBの都合を持たず、API仕様（リクエスト/レスポンス）だけに合わせて設計します。不変にしたいことが多いので `record` との相性が良いです。

```java
public record UserResponse(Long id, String name, String email) {
    public static UserResponse from(User user) {
        return new UserResponse(user.getId(), user.getName(), user.getEmail());
    }
}
```

### なぜ Entity をそのまま返さないのか

1. **不要な情報の露出** — `password` のような内部フィールドまでJSONに含まれてしまう危険がある
2. **無限参照** — `User → Order → User → ...` のように双方向関係をそのままJSON化すると循環参照でエラーになる
3. **N+1問題** — 遅延ロード（`LAZY`）のコレクションにControllerからアクセスすると、意図しないタイミングで追加クエリが発生する
4. **API仕様とDBスキーマの結合** — Entityのフィールド名を変えるとAPIレスポンスまで変わってしまい、DBのリファクタリングがしづらくなる

### 層ごとの役割分担

```
Controller  ← DTO  →  クライアント
    ↓ ↑
Service     ← DTO / Entity 変換 →
    ↓ ↑
Repository  ← Entity →  DB
```

- Controllerの入出力は常にDTO（Request/Response）
- Serviceの中でEntityとDTOを変換する
- RepositoryはEntityのみを扱う

```java
@Service
public class UserService {
    private final UserRepository userRepository;

    public UserResponse getUser(Long id) {
        User user = userRepository.findById(id)
                .orElseThrow(() -> new IllegalArgumentException("존재하지 않는 사용자"));
        return UserResponse.from(user); // Entity → DTO
    }
}
```

### まとめ

| 項目 | Entity | DTO |
|---|---|---|
| 目的 | DBテーブルの永続化 | 層間のデータ転送 |
| 識別子 | あり（`@Id`） | 基本的になし |
| 可変性 | 可変（ダーティチェッキング） | 不変が望ましい（`record`） |
| 関係 | 他Entityと連関（`@OneToMany`等） | 必要な値のみ平坦に持つ |
| 露出範囲 | Repository/Service内部 | Controller ↔ クライアント |

### Entity란?

`@Entity`는 DB 테이블과 1:1로 매핑되는 객체입니다. JPA(Hibernate)의 영속성 컨텍스트가 상태를 관리하며, 필드를 변경하면 그대로 UPDATE 쿼리로 반영되기도 합니다(더티 체킹).

```java
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String email;
    private String password; // DB에만 존재해야 하는 값

    @OneToMany(mappedBy = "user")
    private List<Order> orders;
}
```

### DTO란?

DTO(Data Transfer Object)는 계층과 계층 사이에서 데이터를 옮기기 위한 단순한 그릇입니다. DB의 사정을 담지 않고, API 스펙(요청/응답)에만 맞춰 설계합니다. 불변으로 만드는 경우가 많아 `record`와 궁합이 좋습니다.

```java
public record UserResponse(Long id, String name, String email) {
    public static UserResponse from(User user) {
        return new UserResponse(user.getId(), user.getName(), user.getEmail());
    }
}
```

### 왜 Entity를 그대로 반환하면 안 되는가

1. **불필요한 정보 노출** — `password` 같은 내부 필드까지 JSON에 포함될 위험이 있음
2. **무한 참조** — `User → Order → User → ...` 처럼 양방향 관계를 그대로 JSON으로 직렬화하면 순환 참조로 에러가 발생함
3. **N+1 문제** — 지연 로딩(`LAZY`) 컬렉션에 Controller에서 접근하면 의도치 않은 시점에 추가 쿼리가 발생함
4. **API 스펙과 DB 스키마의 결합** — Entity 필드명을 바꾸면 API 응답까지 함께 바뀌어버려서 DB 리팩터링이 어려워짐

### 계층별 역할 분담

```
Controller  ← DTO  →  클라이언트
    ↓ ↑
Service     ← DTO / Entity 변환 →
    ↓ ↑
Repository  ← Entity →  DB
```

- Controller의 입출력은 항상 DTO(Request/Response)
- Service 내부에서 Entity와 DTO를 서로 변환
- Repository는 Entity만 다룸

```java
@Service
public class UserService {
    private final UserRepository userRepository;

    public UserResponse getUser(Long id) {
        User user = userRepository.findById(id)
                .orElseThrow(() -> new IllegalArgumentException("존재하지 않는 사용자"));
        return UserResponse.from(user); // Entity → DTO
    }
}
```

### 정리

| 항목 | Entity | DTO |
|---|---|---|
| 목적 | DB 테이블 영속화 | 계층 간 데이터 전달 |
| 식별자 | 있음(`@Id`) | 기본적으로 없음 |
| 가변성 | 가변(더티 체킹) | 불변이 바람직함(`record`) |
| 관계 | 다른 Entity와 연관(`@OneToMany` 등) | 필요한 값만 평평하게 보유 |
| 노출 범위 | Repository/Service 내부 | Controller ↔ 클라이언트 |
