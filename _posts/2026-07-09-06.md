### 全体の流れ

Spring Bootはブラウザからの要求を受けてDBとやり取りし、また応答を返すサーバープログラムです。処理は3つの層に分かれています。

```text
ブラウザ
   ↓
Controller  (要求を受け取る)
   ↓
Service     (業務ロジックを実行する)
   ↓
Repository  (DBにアクセスする)
   ↓
Database
```

### レストランに例えると

- **Controller = ホールスタッフ**：注文を受けて伝えるだけで、料理はしない
- **Service = シェフ**：実際に調理する、つまり「業務ロジック」を実行する（会員登録、ログイン、パスワード変更、ポイント計算など）
- **Repository = 倉庫係**：食材（データ）を出し入れするだけで、料理（ロジック）はしない

```java
@RestController
public class UserController {
    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping("/users/{id}")
    public UserResponse getUser(@PathVariable Long id) {
        return userService.getUser(id);
    }
}

@Service
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public UserResponse getUser(Long id) {
        User user = userRepository.findById(id)
                .orElseThrow(() -> new IllegalArgumentException("존재하지 않는 사용자"));
        return UserResponse.from(user);
    }
}

@Repository
public class UserRepository {
    // DBからの検索・保存・削除だけを担当
}
```

### DIがなかったら?

各クラスが必要なオブジェクトを自分で `new` するとどうなるでしょうか。

```java
public class UserController {
    UserService userService = new UserService(new UserRepository());
}
```

一見動きそうに見えますが、問題があります。テストのときは本物のDBを使いたくないので `FakeUserRepository` を代わりに使いたいのに、`UserController` の中に `new UserService(new UserRepository())` と直接書かれていると差し替える方法がありません。ControllerとServiceが**強く結合**してしまっているためです。

### DIとは何か

Springはプログラムの起動時に `Controller` / `Service` / `Repository` のオブジェクトをあらかじめ全部作っておき、コンストラクタを通じて互いに接続します。

```java
@RestController
public class UserController {
    private final UserService userService;

    public UserController(UserService userService) { // Springが自動で渡す
        this.userService = userService;
    }
}
```

自分で `new` する代わりに、Springが必要なオブジェクトを見つけて生成し、必要な場所に注入してくれる仕組みを **DI（Dependency Injection、依存性注入）** と呼びます。テストのときはこの注入口に `FakeUserRepository` を差し込めるため、強い結合の問題も解消されます。

### なぜコンストラクタ注入を使うのか

昔はフィールドに直接注入する方式（`@Autowired`）がよく使われましたが、今はコンストラクタ注入が推奨されます。

1. 必須のオブジェクトが揃わないとオブジェクト自体を生成できず、起動直後に失敗するため問題を早期に発見できる
2. テストで欲しい実装（Fakeなど）をコンストラクタ経由で簡単に差し込める

### アノテーションの意味

`@RestController` / `@Service` / `@Repository` は、Springに「このクラスの役割は何か」を伝える印です。

| アノテーション | 意味 |
|---|---|
| `@RestController` | HTTPリクエストを受け取るクラス |
| `@Service` | 業務ロジックを担当するクラス |
| `@Repository` | DBと通信するクラス |

これらが付いたクラスをSpringが見つけると、そのオブジェクトを自動で生成し、DIで必要な場所に接続してくれます。

### MVCパターンとの関係

MVCはプログラムを役割ごとに分ける設計です。

- **Model**：データと業務ロジック
- **View**：画面表示
- **Controller**：要求を受けてModelとViewをつなぐ

昔ながらのSpring MVCはControllerがModelを処理し、HTMLをViewとして返していました。しかし今日のSpring Boot APIサーバーはReact/VueのようなフロントエンドにJSONだけを返すため、Viewはほとんど見えません。

さらに、元々のModelが会員照会・パスワード検証・DBアクセス・ポイント計算まで全部担当していたため肥大化しやすく、実務ではModelを`Service`・`Repository`・`Entity`に細分化しました。

| MVC | Spring Boot |
|---|---|
| View | ブラウザ（React、HTMLなど） |
| Controller | Controller |
| Model | Service + Repository + Entity |

つまりController-Service-Repositoryという3層構造は、MVCを実務で保守しやすいように発展させた形だと理解すればよいです。

### 전체 흐름

Spring Boot는 브라우저의 요청을 받아 DB와 주고받은 뒤 다시 응답을 돌려주는 서버 프로그램입니다. 이 처리 과정은 3개의 계층으로 나뉩니다.

```text
브라우저
   ↓
Controller  (요청을 받는다)
   ↓
Service     (비즈니스 로직을 실행한다)
   ↓
Repository  (DB에 접근한다)
   ↓
Database
```

### 레스토랑에 비유하면

- **Controller = 홀 직원**: 주문을 받아 전달만 할 뿐, 요리는 하지 않음
- **Service = 주방장**: 실제로 요리, 즉 "비즈니스 로직"을 수행함 (회원가입, 로그인, 비밀번호 변경, 포인트 계산 등)
- **Repository = 창고 담당**: 재료(데이터)를 꺼내고 넣을 뿐, 요리(로직)는 하지 않음

```java
@RestController
public class UserController {
    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping("/users/{id}")
    public UserResponse getUser(@PathVariable Long id) {
        return userService.getUser(id);
    }
}

@Service
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public UserResponse getUser(Long id) {
        User user = userRepository.findById(id)
                .orElseThrow(() -> new IllegalArgumentException("존재하지 않는 사용자"));
        return UserResponse.from(user);
    }
}

@Repository
public class UserRepository {
    // DB 조회/저장/삭제만 담당
}
```

### DI가 없다면?

각 클래스가 필요한 객체를 스스로 `new` 한다면 어떻게 될까요.

```java
public class UserController {
    UserService userService = new UserService(new UserRepository());
}
```

당장은 문제없어 보이지만, 테스트할 때 진짜 DB를 쓰고 싶지 않아 `FakeUserRepository`로 바꿔치기하고 싶어도 `UserController` 안에 `new UserService(new UserRepository())`가 직접 박혀 있으면 교체할 방법이 없습니다. Controller와 Service가 **강하게 결합(Tight Coupling)** 되어 있기 때문입니다.

### DI란 무엇인가

Spring은 프로그램이 시작될 때 `Controller` / `Service` / `Repository` 객체를 미리 전부 만들어두고, 생성자를 통해 서로 연결해줍니다.

```java
@RestController
public class UserController {
    private final UserService userService;

    public UserController(UserService userService) { // Spring이 자동으로 넣어줌
        this.userService = userService;
    }
}
```

내가 직접 `new`로 객체를 만드는 대신, Spring이 필요한 객체를 찾아 생성하고 필요한 곳에 꽂아주는 방식을 **DI(Dependency Injection, 의존성 주입)**라고 부릅니다. 테스트할 때는 이 주입 지점에 `FakeUserRepository`를 끼워 넣을 수 있어 강한 결합 문제도 해결됩니다.

### 왜 생성자 주입을 사용하는가

예전에는 필드에 직접 주입하는 방식(`@Autowired`)이 많이 쓰였지만, 지금은 생성자 주입이 권장됩니다.

1. 필수 객체가 갖춰지지 않으면 객체 자체를 생성할 수 없어 실행 직후 바로 실패하므로 문제를 빠르게 발견할 수 있음
2. 테스트에서 원하는 구현체(Fake 등)를 생성자를 통해 쉽게 끼워 넣을 수 있음

### 어노테이션의 의미

`@RestController` / `@Service` / `@Repository`는 Spring에게 "이 클래스의 역할이 무엇인지" 알려주는 표시입니다.

| 어노테이션 | 의미 |
|---|---|
| `@RestController` | HTTP 요청을 받는 클래스 |
| `@Service` | 비즈니스 로직을 담당하는 클래스 |
| `@Repository` | DB와 통신하는 클래스 |

이 표시가 붙은 클래스를 Spring이 발견하면 해당 객체를 자동으로 생성하고, DI로 필요한 곳에 연결해줍니다.

### MVC 패턴과의 관계

MVC는 프로그램을 역할별로 나누는 설계 방식입니다.

- **Model**: 데이터와 비즈니스 로직
- **View**: 화면 표시
- **Controller**: 요청을 받아 Model과 View를 연결

예전 Spring MVC는 Controller가 Model을 처리하고 HTML을 View로 반환했습니다. 하지만 오늘날의 Spring Boot API 서버는 React/Vue 같은 프론트엔드에 JSON만 반환하기 때문에 View가 거의 보이지 않습니다.

게다가 원래 Model이 회원 조회·비밀번호 검증·DB 접근·포인트 계산까지 전부 담당하다 보니 쉽게 비대해졌고, 실무에서는 Model을 `Service`·`Repository`·`Entity`로 세분화했습니다.

| MVC | Spring Boot |
|---|---|
| View | 브라우저(React, HTML 등) |
| Controller | Controller |
| Model | Service + Repository + Entity |

즉 Controller-Service-Repository라는 3계층 구조는 MVC를 실무에서 유지보수하기 쉽도록 발전시킨 형태라고 이해하면 됩니다.
