### record とは

`record` は、不変（immutable）のデータを保持するためのクラスを簡潔に定義できる機能です（Java 16で正式導入）。フィールド、コンストラクタ、`equals`/`hashCode`/`toString`、各フィールドのアクセサをコンパイラが自動生成してくれるため、単純にデータをまとめるだけのクラス（DTOやValue Object）を書くボイラープレートが大幅に減ります。

```java
public record Point(int x, int y) {}

Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);

p1.x();          // 1（アクセサは getX() ではなく x()）
p1.equals(p2);   // true（フィールドの値で比較）
p1.toString();   // "Point[x=1, y=2]"
```

### 値が同じなら同じオブジェクト？（equals と ==）

`record` は値が同じであれば `equals()` が `true` を返すように自動生成されます。

```java
Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);

System.out.println(p1.equals(p2)); // true
System.out.println(p1 == p2);      // false
```

- `equals()` はフィールドの値を比較するので `true`
- `==` はメモリ上の同一オブジェクトかどうかを比較するので `false`（`p1` と `p2` は別々に生成されたオブジェクト）
- 通常のクラスでは `equals()` を自分でオーバーライドしない限り `==` と同じ挙動になる

- フィールドは `private final` として自動生成され、外部からは変更できない
- コンパクトコンストラクタでバリデーションを追加できる

```java
public record Point(int x, int y) {
    public Point {
        if (x < 0 || y < 0) {
            throw new IllegalArgumentException("負の値は許可されません");
        }
    }
}
```

- 継承はできない（`record` は暗黙的に `final`）が、インターフェースの実装は可能

### なぜ不変（immutable）が嬉しいのか

一度作った値が途中で書き換えられると、思わぬバグの原因になります。

- 会員番号のような重要な値が、コードのどこかで意図せず変更されてしまう
- APIレスポンス用に作ったデータを、別の処理が途中で書き換えてしまう

このような事故を防ぐため、`record` は一度作ったら値を変更できないようになっています。

```java
Point p = new Point(1, 2);
// p.x = 10; // コンパイルエラー：フィールドを直接変更できない
```

### DTOとの相性

DTO（Data Transfer Object）とは、データを運ぶためだけのシンプルな入れ物です。例えば会員情報を返すレスポンスは、名前・年齢だけを持てば十分で、特別なロジックは必要ありません。

```java
public record UserResponse(String name, int age) {}
```

このように、計算やロジックをほとんど持たず、ただデータを保持するだけのオブジェクトには `record` がよく合います。

### なぜ JPA の `@Entity` には `record` を使えないのか

Spring の JPA（Hibernate）は、データベースのテーブルをオブジェクトとして扱うために `@Entity` を使います。

```java
@Entity
class User {
    private Long id;
    private String name;
}
```

JPAはこのオブジェクトを次のように操作します。

1. 空のオブジェクトを生成する
2. フィールドに値を一つずつ詰める
3. 必要に応じてプロキシオブジェクトを作る

しかし `record` は

- すべてのフィールドが `final` で後から値を変更できない
- 他のクラスから継承されることを前提としていない

という性質を持つため、この動作方式と噛み合いません。そのため、役割ごとに次のように使い分けるのが一般的です。

| 用途 | 使用する型 |
|---|---|
| データベースと連携するオブジェクト | 通常のクラス（`class`） |
| APIリクエスト/レスポンスなどデータの受け渡し用オブジェクト | `record` |

### var とは

`var` はローカル変数の型推論を行うキーワードです（Java 10で導入）。右辺の式からコンパイラが型を推論するため、冗長な型宣言を省略できます。あくまでコンパイル時に型が確定する「静的型付け」であり、動的型付けではありません。

```java
var list = new ArrayList<String>(); // ArrayList<String> として推論される
var map = new HashMap<String, Integer>();

for (var entry : map.entrySet()) {  // 拡張for文でも使用可能
    System.out.println(entry.getKey());
}
```

- 使えるのはローカル変数・for文の変数・try-with-resourcesの変数のみ
- フィールド、メソッドの引数、戻り値の型には使えない
- 初期化式が必須（`var x;` のような宣言のみはエラー）
- 型が自明でない場合は可読性を下げることもあるため、変数名や右辺から型が明確なときに使うのが望ましい

### var と JavaScript の var は違う

`var` という名前はJavaScriptにもありますが、動作は全く異なります。

```javascript
// JavaScript
var x = 10;
x = "hello"; // 動的型付けなのでOK
```

```java
// Java
var name = "Claude"; // コンパイラが String と推論する
name = 10;            // コンパイルエラー：String型にintは代入できない
```

Javaは静的型付け言語なので、`var` はあくまで「型宣言を省略できる」だけであり、型そのものがなくなるわけではありません。コンパイル後は通常の型宣言と同じように扱われます。

### まとめ

| 機能 | 目的 | 導入バージョン |
|---|---|---|
| `record` | 不変データクラスのボイラープレート削減 | Java 16 |
| `var` | ローカル変数の型推論 | Java 10 |

どちらもJava 21（LTS）でそのまま使用できる基本機能。

### record란?

`record`는 불변(immutable) 데이터를 담는 클래스를 간결하게 정의할 수 있는 기능입니다 (Java 16에서 정식 도입). 필드, 생성자, `equals`/`hashCode`/`toString`, 각 필드의 접근자를 컴파일러가 자동으로 생성해주기 때문에, 단순히 데이터를 묶기만 하는 클래스(DTO나 Value Object)를 작성할 때의 보일러플레이트가 크게 줄어듭니다.

```java
public record Point(int x, int y) {}

Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);

p1.x();          // 1 (접근자는 getX()가 아니라 x())
p1.equals(p2);   // true (필드 값 기준으로 비교)
p1.toString();   // "Point[x=1, y=2]"
```

### 값이 같으면 같은 객체? (equals와 ==)

`record`는 값이 같으면 `equals()`가 `true`를 반환하도록 자동으로 만들어줍니다.

```java
Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);

System.out.println(p1.equals(p2)); // true
System.out.println(p1 == p2);      // false
```

- `equals()`는 필드 값을 비교하므로 `true`
- `==`는 메모리상 같은 객체인지를 비교하므로 `false` (`p1`과 `p2`는 각각 따로 생성된 객체)
- 일반 클래스는 `equals()`를 직접 오버라이드하지 않는 한 `==`와 같은 결과를 보임

- 필드는 `private final`로 자동 생성되어 외부에서 값을 변경할 수 없음
- 컴팩트 생성자(compact constructor)로 유효성 검사를 추가할 수 있음

```java
public record Point(int x, int y) {
    public Point {
        if (x < 0 || y < 0) {
            throw new IllegalArgumentException("음수는 허용되지 않습니다");
        }
    }
}
```

- 상속은 불가능하지만 (`record`는 암묵적으로 `final`), 인터페이스 구현은 가능

### 왜 불변(immutable)이 좋은가?

한 번 만든 값이 중간에 바뀌면 예상치 못한 버그의 원인이 됩니다.

- 주민등록번호처럼 중요한 값이 코드 어딘가에서 의도치 않게 바뀌어 버리는 경우
- API 응답으로 만든 데이터를 다른 로직이 중간에 바꿔버리는 경우

이런 사고를 막기 위해 record는 한 번 만들면 값을 바꿀 수 없도록 설계되어 있습니다.

```java
Point p = new Point(1, 2);
// p.x = 10; // 컴파일 에러: 필드를 직접 바꿀 수 없음
```

### DTO와의 궁합

DTO(Data Transfer Object)는 데이터를 옮기기 위한 단순한 상자입니다. 예를 들어 회원 정보 응답은 이름, 나이 정도만 담으면 되고 별다른 로직이 필요 없습니다.

```java
public record UserResponse(String name, int age) {}
```

이렇게 계산이나 로직 없이 데이터만 담는 객체에는 record가 잘 어울립니다.

### record는 왜 JPA `@Entity`에는 사용할 수 없을까?

Spring의 JPA(Hibernate)는 데이터베이스 테이블을 객체로 다루기 위해 `@Entity`를 사용합니다.

```java
@Entity
class User {
    private Long id;
    private String name;
}
```

JPA는 이 객체를 다음과 같은 방식으로 다룹니다.

1. 빈 객체를 생성한다
2. 필드에 값을 하나씩 채운다
3. 필요하면 프록시 객체를 만든다

하지만 record는

- 모든 필드가 `final`이라 나중에 값을 바꿀 수 없고
- 다른 클래스가 상속하는 것을 전제로 하지 않음

이런 특성 때문에 JPA의 동작 방식과 맞지 않습니다. 그래서 보통 용도에 따라 다음과 같이 나눠 사용합니다.

| 용도 | 사용 타입 |
|---|---|
| 데이터베이스와 연결되는 객체 | 일반 클래스 (`class`) |
| API 요청/응답처럼 데이터를 전달하는 객체 | `record` |

### var란?

`var`는 로컬 변수의 타입 추론을 위한 키워드입니다 (Java 10에서 도입). 우변의 식으로부터 컴파일러가 타입을 추론하기 때문에 불필요하게 긴 타입 선언을 생략할 수 있습니다. 컴파일 시점에 타입이 확정되는 정적 타입 시스템이며, 동적 타입이 아닙니다.

```java
var list = new ArrayList<String>(); // ArrayList<String>으로 추론됨
var map = new HashMap<String, Integer>();

for (var entry : map.entrySet()) {  // 확장 for문에서도 사용 가능
    System.out.println(entry.getKey());
}
```

- 로컬 변수, for문 변수, try-with-resources 변수에서만 사용 가능
- 필드, 메서드 매개변수, 반환 타입에는 사용 불가
- 초기화식이 반드시 있어야 함 (`var x;`처럼 선언만 하면 컴파일 에러)
- 타입이 명확하지 않은 경우 오히려 가독성을 해칠 수 있으므로, 변수명이나 우변에서 타입이 분명할 때 사용하는 것이 바람직함

### var와 JavaScript의 var는 다르다

이름은 같지만 동작 방식은 완전히 다릅니다.

```javascript
// JavaScript
var x = 10;
x = "hello"; // 동적 타입이라 가능
```

```java
// Java
var name = "Claude"; // 컴파일러가 String으로 추론
name = 10;            // 컴파일 에러: String 타입에 int를 대입할 수 없음
```

Java는 정적 타입 언어이기 때문에 `var`는 "타입 선언을 생략"하는 것일 뿐, 타입 자체가 없어지는 것은 아닙니다. 컴파일이 끝나면 일반적인 타입 선언과 동일하게 동작합니다.

### 정리

| 기능 | 목적 | 도입 버전 |
|---|---|---|
| `record` | 불변 데이터 클래스의 보일러플레이트 감소 | Java 16 |
| `var` | 로컬 변수 타입 추론 | Java 10 |

둘 다 Java 21(LTS)에서 그대로 사용 가능한 기본 기능이다.
