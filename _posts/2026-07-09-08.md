### ORM・JPA・Hibernateの関係

**ORM（Object-Relational Mapping）** は、Javaのクラス（オブジェクト）とDBのテーブル（関係）を対応づける技術全般を指します。SQL文を手で書く代わりに、オブジェクトを扱う感覚でDBを操作できるようにするものです。

**JPA（Java Persistence API）** はそのORMの「標準仕様（インターフェースの集まり）」であり、それ自体は実装を持ちません。**Hibernate** はJPAを実際に実装したライブラリの一つで、Spring Bootが標準で採用しています。関係を図にすると次のようになります。

```text
JPA        ← 「こういう機能を提供します」という仕様（インターフェース）
  ↑ 実装
Hibernate  ← 実際にSQLを生成して実行する実装体
```

JDBCとJDBCドライバの関係に似ていると考えると理解しやすいです。

### Entityとは

`@Entity` を付けたクラスは、DBの1つのテーブルと対応するオブジェクトです。

```java
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String email;
    private String nickname;
}
```

このクラスのインスタンスを保存・取得・削除すると、Hibernateが裏側でINSERT/SELECT/DELETEなどのSQLを自動生成して実行します。

### 永続性コンテキスト（Persistence Context）とは

永続性コンテキストは、`EntityManager` が管理する「1次キャッシュ」です。取得・保存したエンティティを一時的に保持しておく箱だと考えてください。

```java
@Transactional
public void example(Long id) {
    User user1 = userRepository.findById(id).get(); // ① DBにSELECTクエリが発行される
    User user2 = userRepository.findById(id).get(); // ② SQLは発行されず、①でキャッシュされたuser1がそのまま返る

    System.out.println(user1 == user2); // true（同じインスタンス）
}
```

同じトランザクションの中で同じidを2回取得しても、実際のSQLは1回しか発行されません。2回目は永続性コンテキストにキャッシュされたオブジェクトがそのまま返されます。

#### ダーティチェッキング（Dirty Checking）

さらに重要な特徴として、エンティティのフィールド値を変更するだけで、明示的に保存メソッドを呼ばなくても、トランザクションが終わるタイミングで変更が自動検知され、UPDATE文が発行されます。

```java
@Transactional
public void updateNickname(Long id, String newNickname) {
    User user = userRepository.findById(id).get();
    user.setNickname(newNickname); // フィールドを変更しただけ

    // userRepository.save(user); ← 呼ばなくてもOK
    // トランザクションがコミットされる瞬間、Hibernateが変更を検知してUPDATEを発行する
}
```

なぜこれが可能かというと、永続性コンテキストは各エンティティを取得した瞬間のスナップショット（コピー）を保持しており、トランザクション終了時に「現在の状態」と「スナップショット」を比較して差分があればUPDATEクエリを組み立てるためです。

### 連関関係（Association Mapping）

`@ManyToOne` / `@OneToMany` のようなアノテーションで、テーブル間の外部キー関係をオブジェクトのグラフとして表現します。例えば「1人のユーザーは複数の投稿を持つ」という関係は次のように書きます。

```java
@Entity
public class Post {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User author;
}

@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @OneToMany(mappedBy = "author")
    private List<Post> posts;
}
```

- `@ManyToOne`：多くのPostが1人のUserに属する側（外部キーを持つ側）に付ける
- `@OneToMany(mappedBy = "author")`：関係の「持ち主」ではない側に付け、相手側のフィールド名（`author`）を指定する

#### EAGERとLAZY

関連するエンティティを「いつ読み込むか」を指定できます。

```java
@ManyToOne(fetch = FetchType.EAGER) // デフォルト（@ManyToOneの場合）
@ManyToOne(fetch = FetchType.LAZY)  // 明示的に遅延読み込みを指定
```

- **EAGER（即時読み込み）**：エンティティを取得した瞬間、関連エンティティも一緒にSELECTする
- **LAZY（遅延読み込み）**：実際にそのフィールドにアクセスした瞬間にSELECTする（プロキシオブジェクトが差し込まれている）

`@ManyToOne`のデフォルトはEAGERですが、不要な関連まで毎回読み込むと性能が悪化するため、実務では明示的にLAZYを指定することが多いです。

#### LazyInitializationExceptionに注意

LAZYで設定したフィールドは、永続性コンテキストが閉じた後（＝トランザクションが終わった後）にアクセスすると例外が発生します。

```java
@Transactional
public User getUser(Long id) {
    return userRepository.findById(id).get(); // ここまではトランザクション内
}

// トランザクションの外（Controllerなど）で
User user = userService.getUser(1L);
user.getPosts().size(); // LazyInitializationException！
```

`getUser` メソッドが終わった時点でトランザクション（＝永続性コンテキスト）が閉じられるため、その後で `posts` のようなLAZYフィールドにアクセスしようとしてもDBへの追加問い合わせができず、例外になります。対策としては、必要なデータをトランザクション内で先に取得しておく、あるいはDTOに変換してから返す、といった方法が使われます。

### Repositoryとは

`JpaRepository<エンティティ型, ID型>` を継承するだけで、`save` / `findById` / `findAll` / `delete` のようなCRUDメソッドを、Spring が実行時に自動で実装してくれます。

```java
public interface UserRepository extends JpaRepository<User, Long> {
}
```

インターフェースの中身は空でも、実際には裏でHibernateを使ったSQL発行が全部行われます。さらにメソッド名から自動でクエリを組み立てる機能もあります。

```java
public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByEmail(String email); // WHERE email = ? を自動生成
}
```

### まとめ

| 用語 | 意味 |
|---|---|
| ORM | オブジェクトとテーブルを対応づける技術全般 |
| JPA | ORMの標準仕様（インターフェース） |
| Hibernate | JPAの実装体、Spring Bootが標準採用 |
| 永続性コンテキスト | EntityManagerが管理する1次キャッシュ、ダーティチェッキングもここで行われる |
| EAGER / LAZY | 関連エンティティを即時に読むか、必要な時だけ読むか |
| Repository | `JpaRepository` を継承するだけでCRUDが自動実装される |

### ORM · JPA · Hibernate의 관계

**ORM(Object-Relational Mapping)** 은 자바 클래스(객체)와 DB 테이블(관계형 데이터)을 서로 대응시켜주는 기술 전반을 말합니다. SQL을 직접 손으로 작성하는 대신, 객체를 다루는 감각으로 DB를 조작할 수 있게 해줍니다.

**JPA(Java Persistence API)** 는 이 ORM의 "표준 명세(인터페이스 모음)"이고, 그 자체로는 구현체를 가지고 있지 않습니다. **Hibernate** 는 JPA를 실제로 구현한 라이브러리 중 하나이며, Spring Boot가 기본으로 채택하고 있습니다. 관계를 그림으로 나타내면 다음과 같습니다.

```text
JPA        ← "이런 기능을 제공한다"는 명세(인터페이스)
  ↑ 구현
Hibernate  ← 실제로 SQL을 생성하고 실행하는 구현체
```

JDBC와 JDBC 드라이버의 관계와 비슷하다고 생각하면 이해하기 쉽습니다.

### Entity란?

`@Entity`를 붙인 클래스는 DB의 테이블 하나와 대응되는 객체입니다.

```java
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String email;
    private String nickname;
}
```

이 클래스의 인스턴스를 저장·조회·삭제하면, Hibernate가 내부적으로 INSERT/SELECT/DELETE 같은 SQL을 자동으로 생성해서 실행합니다.

### 영속성 컨텍스트(Persistence Context)란?

영속성 컨텍스트는 `EntityManager`가 관리하는 "1차 캐시"입니다. 조회·저장한 엔티티를 잠시 담아두는 상자라고 생각하면 됩니다.

```java
@Transactional
public void example(Long id) {
    User user1 = userRepository.findById(id).get(); // ① DB에 SELECT 쿼리가 나감
    User user2 = userRepository.findById(id).get(); // ② SQL이 나가지 않고, ①에서 캐시된 user1이 그대로 반환됨

    System.out.println(user1 == user2); // true (같은 인스턴스)
}
```

같은 트랜잭션 안에서 같은 id로 두 번 조회해도, 실제 SQL은 한 번만 나갑니다. 두 번째 조회는 영속성 컨텍스트에 캐시된 객체가 그대로 반환되는 것입니다.

#### 더티 체킹(Dirty Checking)

더 중요한 특징으로, 엔티티의 필드 값만 바꿔주면 저장 메서드를 명시적으로 호출하지 않아도 트랜잭션이 끝나는 시점에 변경 사항이 자동으로 감지되어 UPDATE문이 나갑니다.

```java
@Transactional
public void updateNickname(Long id, String newNickname) {
    User user = userRepository.findById(id).get();
    user.setNickname(newNickname); // 필드만 바꿨을 뿐

    // userRepository.save(user); ← 호출하지 않아도 됨
    // 트랜잭션이 커밋되는 순간, Hibernate가 변경을 감지해서 UPDATE를 날림
}
```

이게 가능한 이유는, 영속성 컨텍스트가 엔티티를 조회한 순간의 스냅샷(복사본)을 함께 보관하고 있다가, 트랜잭션이 끝날 때 "현재 상태"와 "스냅샷"을 비교해서 차이가 있으면 UPDATE 쿼리를 만들어내기 때문입니다.

### 연관관계 매핑(Association Mapping)

`@ManyToOne` / `@OneToMany` 같은 어노테이션으로 테이블 간의 외래 키 관계를 객체 그래프로 표현합니다. 예를 들어 "한 명의 유저는 여러 개의 게시글을 가진다"는 관계는 다음과 같이 작성합니다.

```java
@Entity
public class Post {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User author;
}

@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @OneToMany(mappedBy = "author")
    private List<Post> posts;
}
```

- `@ManyToOne`: 여러 개의 Post가 한 명의 User에 속하는 쪽(외래 키를 가진 쪽)에 붙임
- `@OneToMany(mappedBy = "author")`: 관계의 "주인"이 아닌 쪽에 붙이고, 상대방 필드 이름(`author`)을 지정함

#### EAGER와 LAZY

연관된 엔티티를 "언제 읽어올지"를 지정할 수 있습니다.

```java
@ManyToOne(fetch = FetchType.EAGER) // 기본값(@ManyToOne의 경우)
@ManyToOne(fetch = FetchType.LAZY)  // 명시적으로 지연 로딩 지정
```

- **EAGER(즉시 로딩)**: 엔티티를 조회하는 순간, 연관된 엔티티도 함께 SELECT함
- **LAZY(지연 로딩)**: 실제로 해당 필드에 접근하는 순간에 SELECT함 (프록시 객체가 대신 끼워져 있음)

`@ManyToOne`의 기본값은 EAGER지만, 불필요한 연관까지 매번 읽어오면 성능이 나빠지기 때문에 실무에서는 명시적으로 LAZY를 지정하는 경우가 많습니다.

#### LazyInitializationException 주의하기

LAZY로 설정한 필드는 영속성 컨텍스트가 닫힌 뒤(=트랜잭션이 끝난 뒤)에 접근하면 예외가 발생합니다.

```java
@Transactional
public User getUser(Long id) {
    return userRepository.findById(id).get(); // 여기까지는 트랜잭션 내부
}

// 트랜잭션 밖(Controller 등)에서
User user = userService.getUser(1L);
user.getPosts().size(); // LazyInitializationException!
```

`getUser` 메서드가 끝나는 시점에 트랜잭션(=영속성 컨텍스트)이 닫히기 때문에, 그 이후에 `posts` 같은 LAZY 필드에 접근하려고 하면 DB에 추가로 물어볼 방법이 없어서 예외가 발생합니다. 대책으로는 필요한 데이터를 트랜잭션 안에서 미리 조회해두거나, DTO로 변환한 뒤 반환하는 방법이 흔히 쓰입니다.

### Repository란?

`JpaRepository<엔티티타입, ID타입>`을 상속하기만 하면 `save` / `findById` / `findAll` / `delete` 같은 CRUD 메서드를 Spring이 런타임에 자동으로 구현해줍니다.

```java
public interface UserRepository extends JpaRepository<User, Long> {
}
```

인터페이스 본문이 비어 있어도, 실제로는 내부에서 Hibernate를 통한 SQL 실행이 전부 이루어집니다. 여기에 더해 메서드 이름만으로 쿼리를 자동으로 만들어주는 기능도 있습니다.

```java
public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByEmail(String email); // WHERE email = ? 를 자동 생성
}
```

### 정리

| 용어 | 의미 |
|---|---|
| ORM | 객체와 테이블을 대응시키는 기술 전반 |
| JPA | ORM의 표준 명세(인터페이스) |
| Hibernate | JPA의 구현체, Spring Boot가 기본 채택 |
| 영속성 컨텍스트 | EntityManager가 관리하는 1차 캐시, 더티 체킹도 여기서 이뤄짐 |
| EAGER / LAZY | 연관 엔티티를 즉시 읽을지, 필요할 때만 읽을지 |
| Repository | `JpaRepository`만 상속하면 CRUD가 자동으로 구현됨 |
