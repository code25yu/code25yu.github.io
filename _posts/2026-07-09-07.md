### PostgreSQLとは

PostgreSQLはオープンソースのオブジェクト関係データベース（ORDBMS）です。標準SQLへの準拠度が高く、ACIDトランザクション、外部キー、ビュー、トリガー、ストアドプロシージャなど、業務システムに必要な機能をほぼすべて備えています。

| 用語 | やさしい意味 |
|---|---|
| 標準SQL | 他のDBと似たSQL文法をきちんと守っている |
| ACIDトランザクション | 作業が途中で失敗したら全部元に戻して、データが壊れないようにする |
| 外部キー（Foreign Key） | テーブル同士をつなげて、間違ったデータが入るのを防ぐ |
| ビュー（View） | よく使う検索結果を、まるで仮想のテーブルのように使える |
| トリガー（Trigger） | 特定の出来事が起きたら、DBが自動で別の処理を行う |
| ストアドプロシージャ（Stored Procedure） | DBの中にあらかじめ作っておいたプログラム（関数）のように実行できる |

### 他のデータベースとの違い

| 項目 | PostgreSQL | MySQL | SQLite |
|---|---|---|---|
| アーキテクチャ | クライアント/サーバー | クライアント/サーバー | サーバーレス（ファイル1つで完結） |
| データ型 | 配列、JSONB、範囲型など多彩 | 比較的シンプル | 動的型付け（型チェックが緩い） |
| 同時実行制御 | MVCC（後述） | ストレージエンジンにより異なる（InnoDBはMVCC） | ファイルロック方式、同時書き込みに弱い |
| ライセンス | PostgreSQL License（緩やか） | GPL / 商用 | パブリックドメイン |
| 得意分野 | 複雑なクエリ・データ整合性重視 | Web全般・読み取り中心 | モバイルアプリ・組み込み・テスト用途 |

その他によく比較されるものとして、**MariaDB**（MySQLからフォークされた互換DB）、**Oracle Database**・**SQL Server**（エンタープライズ向け商用RDBMS、堅牢なサポートと引き換えにライセンス費用が高い）、**MongoDB**（テーブルではなくJSON文書を保存するNoSQL、スキーマが固定されない代わりにJOINやトランザクションの扱いはRDBMSほど強力ではない）などがあります。業務システムで整合性の高いリレーショナルデータを扱うならPostgreSQL/MySQL、単一ファイルで完結させたいならSQLite、スキーマが頻繁に変わる/柔軟なドキュメント構造が欲しいならMongoDB、というのが大まかな使い分けの目安です。

### MVCC（多版同時実行制御）とは

行を直接書き換えるのではなく、更新のたびに新しいバージョンの行を作成し、トランザクションごとに「見えるべきバージョン」を判断する方式です。

```sql
-- トランザクションA
BEGIN;
SELECT * FROM accounts WHERE id = 1; -- 残高を確認

-- 同時にトランザクションBが更新してCOMMITしても
-- トランザクションAは自分が開始した時点のスナップショットを見続ける
COMMIT;
```

これにより、読み取りが書き込みをブロックせず（逆も同様）、高い同時実行性能を実現します。ただし、古いバージョンの行（デッドタプル）が溜まるため、`VACUUM` による定期的な整理が必要です。

### 代表的な機能

```sql
-- JSONBカラム: JSONをバイナリ形式で保存し、インデックスも張れる
CREATE TABLE events (
    id SERIAL PRIMARY KEY,
    payload JSONB
);

CREATE INDEX idx_events_payload ON events USING GIN (payload);

SELECT * FROM events WHERE payload @> '{"type": "signup"}';
```

- **JSONB**: JSON文書をそのまま保存しつつ、通常のカラムのようにインデックス・検索が可能
- **配列型**: `text[]` のように1カラムに複数の値を保持できる
- **ウィンドウ関数**: `RANK() OVER (PARTITION BY ...)` のような集計をグループ内で行える
- **CTE（`WITH`句）**: 複雑なクエリを読みやすい単位に分割できる

```sql
WITH recent_orders AS (
    SELECT user_id, COUNT(*) AS order_count
    FROM orders
    WHERE created_at > now() - interval '30 days'
    GROUP BY user_id
)
SELECT u.name, r.order_count
FROM users u
JOIN recent_orders r ON r.user_id = u.id
ORDER BY r.order_count DESC;
```

### トランザクション分離レベル

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

- `READ COMMITTED`（デフォルト）: 各クエリごとに最新のコミット済みデータを見る
- `REPEATABLE READ`: トランザクション開始時点のスナップショットを最後まで維持
- `SERIALIZABLE`: トランザクションを順番に実行したかのように保証する、最も厳格なレベル

### Springで使うときのポイント

Spring BootのJPA（Hibernate）からPostgreSQLを使う場合、`application.yml` にドライバとダイアレクトを指定するだけで、これまで見てきたJSONB・配列型などの拡張機能もほぼそのまま扱えます。

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/mydb
  jpa:
    properties:
      hibernate:
        dialect: org.hibernate.dialect.PostgreSQLDialect
```

### PostgreSQL이란?

PostgreSQL은 오픈소스 객체관계형 데이터베이스(ORDBMS)입니다. 표준 SQL 준수도가 높고, ACID 트랜잭션, 외래 키, 뷰, 트리거, 저장 프로시저 등 업무 시스템에 필요한 기능을 거의 다 갖추고 있습니다.

| 용어 | 쉬운 의미 |
|---|---|
| 표준 SQL | 다른 DB와 비슷한 SQL 문법을 잘 지킨다 |
| ACID 트랜잭션 | 작업이 중간에 실패하면 모두 원래대로 되돌려 데이터가 꼬이지 않게 한다 |
| 외래 키(Foreign Key) | 테이블끼리 연결하여 잘못된 데이터를 막는다 |
| 뷰(View) | 자주 사용하는 조회 결과를 가상의 테이블처럼 사용할 수 있다 |
| 트리거(Trigger) | 특정 일이 발생하면 DB가 자동으로 다른 작업을 수행한다 |
| 저장 프로시저(Stored Procedure) | DB 안에 미리 만들어 둔 프로그램(함수)처럼 실행할 수 있다 |

### 다른 데이터베이스와의 차이점

| 항목 | PostgreSQL | MySQL | SQLite |
|---|---|---|---|
| 아키텍처 | 클라이언트/서버 | 클라이언트/서버 | 서버리스(파일 하나로 동작) |
| 데이터 타입 | 배열, JSONB, range 타입 등 다양 | 비교적 단순 | 동적 타입(타입 검사가 느슨함) |
| 동시성 제어 | MVCC(후술) | 스토리지 엔진에 따라 다름(InnoDB는 MVCC) | 파일 잠금 방식, 동시 쓰기에 약함 |
| 라이선스 | PostgreSQL License(느슨함) | GPL / 상용 | 퍼블릭 도메인 |
| 강점 | 복잡한 쿼리·데이터 정합성 중시 | 웹 서비스 전반·읽기 중심 | 모바일 앱·임베디드·테스트 용도 |

그 외 자주 비교되는 것들로는 **MariaDB**(MySQL에서 포크된 호환 DB), **Oracle Database**·**SQL Server**(엔터프라이즈용 상용 RDBMS로, 견고한 지원을 제공하는 대신 라이선스 비용이 높음), **MongoDB**(테이블이 아닌 JSON 문서를 저장하는 NoSQL로, 스키마가 고정되지 않는 대신 JOIN이나 트랜잭션 처리는 RDBMS만큼 강력하지 않음) 등이 있습니다. 정합성이 중요한 관계형 데이터를 다룬다면 PostgreSQL/MySQL, 파일 하나로 끝내고 싶다면 SQLite, 스키마가 자주 바뀌거나 유연한 문서 구조가 필요하다면 MongoDB를 고려하는 것이 대략적인 선택 기준입니다.

### MVCC(다중 버전 동시성 제어)란?

행을 직접 덮어쓰는 대신, 갱신할 때마다 새로운 버전의 행을 만들고 트랜잭션마다 "자신이 봐야 할 버전"을 판단하는 방식입니다.

```sql
-- 트랜잭션 A
BEGIN;
SELECT * FROM accounts WHERE id = 1; -- 잔액 확인

-- 동시에 트랜잭션 B가 갱신하고 COMMIT해도
-- 트랜잭션 A는 자신이 시작한 시점의 스냅샷을 계속 봄
COMMIT;
```

이 덕분에 읽기가 쓰기를 막지 않고(반대도 마찬가지) 높은 동시성을 확보할 수 있습니다. 다만 오래된 버전의 행(dead tuple)이 쌓이기 때문에 `VACUUM`을 통한 주기적인 정리가 필요합니다.

### 대표 기능

```sql
-- JSONB 컬럼: JSON을 바이너리 형태로 저장하고 인덱스도 걸 수 있음
CREATE TABLE events (
    id SERIAL PRIMARY KEY,
    payload JSONB
);

CREATE INDEX idx_events_payload ON events USING GIN (payload);

SELECT * FROM events WHERE payload @> '{"type": "signup"}';
```

- **JSONB**: JSON 문서를 그대로 저장하면서도 일반 컬럼처럼 인덱싱·검색이 가능
- **배열 타입**: `text[]`처럼 한 컬럼에 여러 값을 담을 수 있음
- **윈도우 함수**: `RANK() OVER (PARTITION BY ...)`처럼 그룹 내 집계를 할 수 있음
- **CTE(`WITH`절)**: 복잡한 쿼리를 읽기 쉬운 단위로 나눌 수 있음

```sql
WITH recent_orders AS (
    SELECT user_id, COUNT(*) AS order_count
    FROM orders
    WHERE created_at > now() - interval '30 days'
    GROUP BY user_id
)
SELECT u.name, r.order_count
FROM users u
JOIN recent_orders r ON r.user_id = u.id
ORDER BY r.order_count DESC;
```

### 트랜잭션 격리 수준

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

- `READ COMMITTED`(기본값): 쿼리를 실행할 때마다 최신 커밋된 데이터를 봄
- `REPEATABLE READ`: 트랜잭션 시작 시점의 스냅샷을 끝까지 유지
- `SERIALIZABLE`: 트랜잭션을 순서대로 실행한 것처럼 보장하는 가장 엄격한 수준

### Spring에서 사용할 때 포인트

Spring Boot의 JPA(Hibernate)에서 PostgreSQL을 사용할 때는 `application.yml`에 드라이버와 dialect만 지정하면, 위에서 본 JSONB·배열 타입 같은 확장 기능도 거의 그대로 활용할 수 있습니다.

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/mydb
  jpa:
    properties:
      hibernate:
        dialect: org.hibernate.dialect.PostgreSQLDialect
```
