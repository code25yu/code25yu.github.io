### なぜこの3つを一緒に学ぶのか

Javaでバックエンドを作るとき、次の3つの役割が必要になります。

```text
Maven / Gradle   ← コードをビルドし、依存ライブラリを管理する（ビルドツール）
Tomcat           ← ビルドされたアプリケーションを実際に動かすサーバー
```

MavenとGradleは「同じ役割（ビルドツール）」を持つ二択の選択肢で、Tomcatは全く別の役割（実行環境）を担います。

### Mavenとは

Mavenはビルドツールです。「ビルド」とは、ソースコードをコンパイルし、必要なライブラリ（依存関係）を取ってきて、実行可能な形（jar/war）にパッケージングする一連の作業を指します。

Mavenは `pom.xml`（Project Object Model）というXMLファイルに、必要な依存ライブラリとバージョンを宣言します。

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <version>3.2.0</version>
    </dependency>
</dependencies>
```

これを書くだけで、Mavenは指定されたライブラリを「リポジトリ」（Maven Central など）から自動でダウンロードし、ローカルの `~/.m2/repository` にキャッシュして使います。手動でjarファイルを集めてくる必要がありません。

#### ビルドライフサイクル

Mavenのビルドは、決まった順序のフェーズ（段階）で進みます。

```text
validate → compile → test → package → verify → install → deploy
```

代表的なコマンド：

```bash
mvn compile   # ソースコードをコンパイルする
mvn test      # テストを実行する
mvn package   # jar/warファイルとしてパッケージングする（この時点でcompile/testも実行される）
mvn install   # ローカルリポジトリに成果物をインストールする
```

例えば `mvn package` を実行すると、Mavenは自動的にそれより前のフェーズ（validate、compile、test）もすべて順番に実行してからパッケージングします。

### Gradleとは

GradleもMaven同様にビルドツールですが、設定をXMLではなくGroovyまたはKotlinのコード（DSL）で書きます。

```kotlin
// build.gradle.kts
dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web:3.2.0")
}
```

コードで書けるため条件分岐やカスタム処理を柔軟に書きやすく、また**増分ビルド（incremental build）**やビルドキャッシュにより、変更のなかった部分の再ビルドをスキップできるため、Mavenより高速になることが多いです。

#### Mavenとの比較

| 項目 | Maven | Gradle |
|---|---|---|
| 設定ファイル | `pom.xml`（XML） | `build.gradle` / `build.gradle.kts`（Groovy/Kotlin） |
| 実行単位 | あらかじめ決まったライフサイクル（phase） | 自由に定義できるタスク（task） |
| ビルド速度 | 標準的 | 増分ビルド・キャッシュにより高速な場合が多い |
| 柔軟性 | 設定重視（Convention over Configuration） | コードで書けるため柔軟 |
| 主な採用例 | 昔からの多くのJavaプロジェクト | 近年のSpring Boot・Androidプロジェクトなど |

どちらも「依存関係を管理し、コードをビルドする」という目的は同じで、プロジェクトによってどちらか一方を選んで使います。

### Tomcatとは

Tomcatは**サーブレットコンテナ**（Webアプリケーションサーバーの一種）です。MavenやGradleでビルドされたJavaのWebアプリケーションを、実際にHTTPリクエストを受け付けて動かす役割を持ちます。

```text
ブラウザ → HTTPリクエスト → Tomcat → アプリケーションのコード → レスポンス
```

昔ながらのやり方では、アプリケーションを `.war`（Web Application Archive）ファイルとしてビルドし、別途インストールしたTomcatに配置（デプロイ）して起動していました。

```text
mvn package → app.war 生成
       ↓
Tomcatの webapps/ フォルダに app.war をコピー
       ↓
Tomcatを起動 → アプリケーションが動く
```

### Spring Bootでは何が変わるのか

Spring Bootには**組み込み（embedded）Tomcat**が最初から含まれています。そのため、外部にTomcatを別途インストールして配置する必要がなく、ビルドした `.jar` ファイルを直接実行するだけでアプリケーションが起動します。

```bash
mvn package        # app.jar を生成（Tomcatも中に含まれている）
java -jar app.jar  # これだけでWebサーバーが起動する
```

つまり「Maven/GradleでビルドしたものをTomcatに配置する」という手順が不要になり、jarファイル1つで完結するようになったのが、Spring Bootが人気になった理由の一つです。

### まとめ

| 用語 | 役割 |
|---|---|
| Maven | XML（`pom.xml`）で依存関係を管理し、決まったライフサイクルでビルドするツール |
| Gradle | コード（Groovy/Kotlin DSL）で柔軟に設定でき、増分ビルドで高速化されたビルドツール |
| Tomcat | ビルドされたWebアプリケーションを実際に動かすサーバー（サーブレットコンテナ） |
| Spring Boot | Tomcatを内蔵しているため、jarファイル1つで起動まで完結する |

### 왜 이 셋을 함께 공부하는가

자바로 백엔드를 만들 때는 다음 세 가지 역할이 필요합니다.

```text
Maven / Gradle   ← 코드를 빌드하고, 의존 라이브러리를 관리한다 (빌드 도구)
Tomcat           ← 빌드된 애플리케이션을 실제로 실행하는 서버
```

Maven과 Gradle은 "같은 역할(빌드 도구)"을 하는 양자택일의 선택지이고, Tomcat은 완전히 다른 역할(실행 환경)을 담당합니다.

### Maven이란?

Maven은 빌드 도구입니다. "빌드"란 소스 코드를 컴파일하고, 필요한 라이브러리(의존성)를 가져오고, 실행 가능한 형태(jar/war)로 패키징하는 일련의 작업을 말합니다.

Maven은 `pom.xml`(Project Object Model)이라는 XML 파일에 필요한 의존 라이브러리와 버전을 선언합니다.

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <version>3.2.0</version>
    </dependency>
</dependencies>
```

이렇게 적어두기만 하면, Maven이 지정된 라이브러리를 "저장소"(Maven Central 등)에서 자동으로 다운로드해서 로컬의 `~/.m2/repository`에 캐시해두고 사용합니다. jar 파일을 수동으로 모아올 필요가 없습니다.

#### 빌드 라이프사이클

Maven의 빌드는 정해진 순서의 단계(phase)로 진행됩니다.

```text
validate → compile → test → package → verify → install → deploy
```

대표적인 명령어:

```bash
mvn compile   # 소스 코드를 컴파일한다
mvn test      # 테스트를 실행한다
mvn package   # jar/war 파일로 패키징한다 (이 시점에 compile/test도 함께 실행됨)
mvn install   # 로컬 저장소에 결과물을 설치한다
```

예를 들어 `mvn package`를 실행하면, Maven은 자동으로 그 앞 단계(validate, compile, test)까지 전부 순서대로 실행한 뒤 패키징합니다.

### Gradle이란?

Gradle도 Maven과 마찬가지로 빌드 도구지만, 설정을 XML이 아니라 Groovy 또는 Kotlin 코드(DSL)로 작성합니다.

```kotlin
// build.gradle.kts
dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web:3.2.0")
}
```

코드로 작성하기 때문에 조건 분기나 커스텀 처리를 유연하게 넣기 쉽고, **증분 빌드(incremental build)**와 빌드 캐시를 통해 변경되지 않은 부분의 재빌드를 건너뛸 수 있어 Maven보다 빠른 경우가 많습니다.

#### Maven과의 비교

| 항목 | Maven | Gradle |
|---|---|---|
| 설정 파일 | `pom.xml`(XML) | `build.gradle` / `build.gradle.kts`(Groovy/Kotlin) |
| 실행 단위 | 미리 정해진 라이프사이클(phase) | 자유롭게 정의 가능한 태스크(task) |
| 빌드 속도 | 표준적 | 증분 빌드·캐시 덕분에 빠른 경우가 많음 |
| 유연성 | 설정 중심(Convention over Configuration) | 코드로 작성 가능해 유연함 |
| 주요 사용 사례 | 오래된 다수의 자바 프로젝트 | 최근의 Spring Boot·Android 프로젝트 등 |

둘 다 "의존성을 관리하고 코드를 빌드한다"는 목적은 같고, 프로젝트마다 둘 중 하나를 선택해서 사용합니다.

### Tomcat이란?

Tomcat은 **서블릿 컨테이너**(웹 애플리케이션 서버의 일종)입니다. Maven이나 Gradle로 빌드된 자바 웹 애플리케이션을 실제로 HTTP 요청을 받아 실행하는 역할을 합니다.

```text
브라우저 → HTTP 요청 → Tomcat → 애플리케이션 코드 → 응답
```

예전 방식에서는 애플리케이션을 `.war`(Web Application Archive) 파일로 빌드한 뒤, 별도로 설치한 Tomcat에 배치(배포)해서 실행했습니다.

```text
mvn package → app.war 생성
       ↓
Tomcat의 webapps/ 폴더에 app.war 복사
       ↓
Tomcat 실행 → 애플리케이션이 동작함
```

### Spring Boot에서는 무엇이 달라지는가

Spring Boot에는 **내장(embedded) Tomcat**이 처음부터 포함되어 있습니다. 그래서 외부에 Tomcat을 별도로 설치하고 배치할 필요 없이, 빌드한 `.jar` 파일을 바로 실행하는 것만으로 애플리케이션이 시작됩니다.

```bash
mvn package        # app.jar 생성 (Tomcat도 안에 포함되어 있음)
java -jar app.jar  # 이것만으로 웹 서버가 실행됨
```

즉 "Maven/Gradle로 빌드한 것을 Tomcat에 배치한다"는 절차가 필요 없어지고, jar 파일 하나로 끝나게 된 것이 Spring Boot가 인기를 얻은 이유 중 하나입니다.

### 정리

| 용어 | 역할 |
|---|---|
| Maven | XML(`pom.xml`)로 의존성을 관리하고, 정해진 라이프사이클로 빌드하는 도구 |
| Gradle | 코드(Groovy/Kotlin DSL)로 유연하게 설정할 수 있고, 증분 빌드로 빨라진 빌드 도구 |
| Tomcat | 빌드된 웹 애플리케이션을 실제로 실행하는 서버 (서블릿 컨테이너) |
| Spring Boot | Tomcat을 내장하고 있어서 jar 파일 하나로 실행까지 끝남 |
