### なぜ必要なのか

JPAには `ddl-auto: update` や `create` のように、Entityクラスからスキーマを自動生成・変更してくれる機能があります。開発初期は便利ですが、実務では危険です。

```yaml
# こういう設定に頼ると…
spring:
  jpa:
    hibernate:
      ddl-auto: update
```

- Entityのフィールドを変更・削除するたびに、Hibernateがどんなスキーマ変更を行うか予測しづらい
- ローカル・ステージング・本番でスキーマがそれぞれ違うタイミング・違う内容で変わってしまう可能性がある
- 最悪の場合、本番DBのカラムやテーブルが意図せず削除される

**Flyway** は、スキーマの変更履歴をSQLファイルとして明示的に残し、ローカル・ステージング・本番のすべての環境に「同じ順序」で「同じ変更」を適用してくれるツールです。

### 基本的な仕組み

```text
db/migration/
  V1__create_users_table.sql
  V2__add_email_to_users.sql
  V3__create_posts_table.sql
```

- 各SQLファイルが1つの「変更」を表す
- Spring Boot起動時に、Flywayがこのフォルダを見て、まだ適用されていないファイルを**バージョン順に**実行する
- 一度適用されたファイルは二度と実行されない（適用済みかどうかはDBに記録される）

### 命名規則

```text
V{バージョン}__{説明}.sql
```

- `V` とバージョン番号の間はそのまま（例: `V1`, `V2`）
- バージョン番号と説明の間は**アンダースコア2つ**（`__`）
- 例: `V1__create_users_table.sql`、`V2__add_email_to_users.sql`

```sql
-- V1__create_users_table.sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    nickname VARCHAR(100) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT now()
);
```

このファイルを `src/main/resources/db/migration/` に置くだけで、Spring Bootが起動時に自動的に見つけて適用します（`spring-boot-starter-flyway` あるいは `flyway-core` 依存関係を追加していることが前提です）。

### 検証の仕組み（チェックサム）

Flywayは適用した各マイグレーションファイルの**チェックサム（内容のハッシュ値）**を、DB内の `flyway_schema_history` というテーブルに記録します。

| installed_rank | version | description | checksum | success |
|---|---|---|---|---|
| 1 | 1 | create users table | 123456789 | true |
| 2 | 2 | add email to users | 987654321 | true |

もし既に適用済みの `V1__create_users_table.sql` の中身を後から書き換えると、実際のファイルのチェックサムと、DBに記録されているチェックサムが一致しなくなります。次回の起動時、Flywayはこの不一致を検知して**エラーを出し、起動を止めます**。

```text
FlywayException: Migration checksum mismatch for migration version 1
```

つまり「一度適用したファイルは編集禁止」というルールです。スキーマを直したい場合は、既存のファイルを直接編集するのではなく、新しいバージョンのファイル（例: `V4__fix_users_email_default.sql`）を追加します。

### ddl-autoとの関係

Flywayを導入したプロジェクトでは、スキーマの「唯一の管理者」はFlywayになります。JPA側にスキーマを変更させてしまうと、Flywayが把握していない変更が紛れ込んでしまうため、`ddl-auto` は次のように固定します。

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate
  flyway:
    enabled: true
```

`validate` は、Entityのマッピングと実際のDBスキーマが一致しているかどうかを**チェックするだけ**で、スキーマ自体には一切手を加えません。もし一致していなければ起動時にエラーになるので、「マイグレーションの適用忘れ」にもすぐ気づけます。

### まとめ

| 概念 | 意味 |
|---|---|
| Flyway | SQLファイルでスキーマ変更履歴を管理し、全環境に同じ順序で適用するツール |
| 命名規則 | `V{バージョン}__{説明}.sql`（アンダースコア2つ） |
| 配置場所 | `src/main/resources/db/migration/` |
| flyway_schema_history | 適用済みマイグレーションとそのチェックサムを記録するテーブル |
| チェックサム不一致 | 適用済みファイルを書き換えると起動時にエラーになる → 新しいバージョンを追加する |
| ddl-auto | Flyway導入時は `validate` に固定し、スキーマ変更はFlywayだけに任せる |

### 왜 필요한가

JPA에는 `ddl-auto: update`나 `create`처럼 Entity 클래스로부터 스키마를 자동으로 생성·변경해주는 기능이 있습니다. 개발 초기에는 편리하지만, 실무에서는 위험합니다.

```yaml
# 이런 설정에 의존하면…
spring:
  jpa:
    hibernate:
      ddl-auto: update
```

- Entity의 필드를 변경·삭제할 때마다 Hibernate가 어떤 스키마 변경을 할지 예측하기 어려움
- 로컬·스테이징·운영에서 스키마가 각각 다른 타이밍에, 다른 내용으로 바뀔 수 있음
- 최악의 경우 운영 DB의 컬럼이나 테이블이 의도치 않게 삭제됨

**Flyway**는 스키마 변경 이력을 SQL 파일로 명시적으로 남기고, 로컬·스테이징·운영 등 모든 환경에 "같은 순서"로 "같은 변경"을 적용해주는 도구입니다.

### 기본 동작 방식

```text
db/migration/
  V1__create_users_table.sql
  V2__add_email_to_users.sql
  V3__create_posts_table.sql
```

- 각 SQL 파일이 하나의 "변경 사항"을 나타냄
- Spring Boot가 시작될 때 Flyway가 이 폴더를 확인해서 아직 적용되지 않은 파일을 **버전 순서대로** 실행함
- 한 번 적용된 파일은 다시 실행되지 않음 (적용 여부는 DB에 기록됨)

### 네이밍 규칙

```text
V{버전}__{설명}.sql
```

- `V`와 버전 번호 사이는 그대로 붙임 (예: `V1`, `V2`)
- 버전 번호와 설명 사이는 **언더스코어 2개**(`__`)
- 예: `V1__create_users_table.sql`, `V2__add_email_to_users.sql`

```sql
-- V1__create_users_table.sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    nickname VARCHAR(100) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT now()
);
```

이 파일을 `src/main/resources/db/migration/`에 두기만 하면, Spring Boot가 부팅 시 자동으로 찾아서 적용합니다(`flyway-core` 의존성이 추가되어 있어야 합니다).

### 검증 메커니즘(체크섬)

Flyway는 적용한 각 마이그레이션 파일의 **체크섬(내용의 해시값)**을 DB 안의 `flyway_schema_history`라는 테이블에 기록합니다.

| installed_rank | version | description | checksum | success |
|---|---|---|---|---|
| 1 | 1 | create users table | 123456789 | true |
| 2 | 2 | add email to users | 987654321 | true |

만약 이미 적용된 `V1__create_users_table.sql`의 내용을 나중에 수정하면, 실제 파일의 체크섬과 DB에 기록된 체크섬이 서로 달라집니다. 다음 부팅 시 Flyway는 이 불일치를 감지해서 **에러를 내고 시작을 멈춥니다.**

```text
FlywayException: Migration checksum mismatch for migration version 1
```

즉 "한 번 적용한 파일은 수정 금지"라는 규칙입니다. 스키마를 고치고 싶다면 기존 파일을 직접 편집하는 대신, 새로운 버전의 파일(예: `V4__fix_users_email_default.sql`)을 추가해야 합니다.

### ddl-auto와의 관계

Flyway를 도입한 프로젝트에서는 스키마의 "유일한 관리자"가 Flyway가 됩니다. JPA가 스키마를 직접 바꾸도록 두면 Flyway가 모르는 변경이 섞여버릴 수 있으므로, `ddl-auto`는 다음과 같이 고정합니다.

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate
  flyway:
    enabled: true
```

`validate`는 Entity 매핑과 실제 DB 스키마가 일치하는지 **검사만 할 뿐**, 스키마 자체는 전혀 건드리지 않습니다. 만약 일치하지 않으면 부팅 시 에러가 발생하므로, "마이그레이션 적용을 깜빡한 상황"도 바로 알아챌 수 있습니다.

### 정리

| 개념 | 의미 |
|---|---|
| Flyway | SQL 파일로 스키마 변경 이력을 관리하고, 모든 환경에 같은 순서로 적용하는 도구 |
| 네이밍 규칙 | `V{버전}__{설명}.sql` (언더스코어 2개) |
| 위치 | `src/main/resources/db/migration/` |
| flyway_schema_history | 적용된 마이그레이션과 그 체크섬을 기록하는 테이블 |
| 체크섬 불일치 | 적용된 파일을 수정하면 부팅 시 에러 발생 → 새 버전 파일을 추가해야 함 |
| ddl-auto | Flyway 도입 시 `validate`로 고정하고, 스키마 변경은 Flyway에만 맡김 |
